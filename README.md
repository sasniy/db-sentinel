# 🛡️ DB Sentinel

Мониторинговая система для баз данных (ClickHouse / PostgreSQL / SQLite) с веб-дашбордом,
правилами проверки качества данных и уведомлениями в Telegram / Email / Mattermost.

Pet-project дата-аналитика: следит, чтобы данные в таблицах **всегда были**,
и сразу уведомляет, если что-то пошло не так.

## Возможности

- **Подключения** к ClickHouse, PostgreSQL и SQLite (легко добавить новые — см. `app/connectors/`)
- **6 типов правил проверки**:
  | Тип | Что ловит |
  |---|---|
  | Свежесть данных | Самая новая запись старше N минут — «данные не приехали» |
  | Объём данных | За окно времени строк меньше минимума |
  | NULL в колонке | Доля NULL выше порога |
  | Дубликаты по ключу | Нарушение уникальности ключевых колонок |
  | Аномалия метрики | Метрика отклонилась от истории своих запусков больше чем на N сигм (z-score) |
  | Аномалия по истории таблицы | Метрика по периодам (день/час) прямо из исторических данных таблицы: последний завершённый период сравнивается с историей; работает с первого запуска, умеет сезонность по дню недели |
  | Свой SQL + порог | Любой запрос, возвращающий число, + условие нормы |
- **Встроенный планировщик** (APScheduler): у каждого правила свой интервал
- **Уведомления**: Telegram-бот, Email (SMTP), Mattermost (личные сообщения через API)
  - cooldown между повторными уведомлениями (не спамит)
  - уведомление о восстановлении, когда проверка снова зелёная
- **Свой SQL-источник**: любое «табличное» правило может работать не по всей
  таблице, а по результату твоего SELECT с фильтрами WHERE/PREWHERE —
  удобно для больших таблиц ClickHouse
- **Веб-дашборд**: статусы, счётчики, история проверок с типом запуска
  (авто/ручной), даты последних запусков обоих типов и время следующего
  автозапуска, кнопка «Запустить все», управление правилами и подключениями
- **Два языка интерфейса**: RU / EN (переключается на лету, хранится в настройках)

## Быстрый старт

```powershell
cd db-sentinel
python -m venv .venv
.venv\Scripts\pip install -r requirements.txt

# (необязательно) демо-база с "испорченными" данными + готовые правила
.venv\Scripts\python demo\make_demo.py

.venv\Scripts\python run.py          # http://127.0.0.1:8000
.venv\Scripts\python run.py 8765     # свой порт
```

Открой http://127.0.0.1:8000 — на дашборде уже будут демо-правила
(свежесть, NULL и дубликаты специально «сломаны», чтобы показать тревоги).

## Настройка уведомлений (вкладка «Настройки»)

**Telegram**: создай бота через [@BotFather](https://t.me/BotFather), возьми токен.
Напиши боту что-нибудь, потом узнай свой chat_id через `https://api.telegram.org/bot<TOKEN>/getUpdates`.

**Email**: SMTP-хост/порт, логин и пароль (для Gmail — «пароль приложения»), список получателей.

**Mattermost**: URL сервера, Personal Access Token (Профиль → Безопасность → Токены доступа)
и получатель — его user ID или `@username`. Бот сам создаст личный диалог и напишет туда.

У каждого канала есть кнопка «Отправить тестовое сообщение».

## Как это устроено

```
db-sentinel/
├── run.py                  # запуск (uvicorn)
├── app/
│   ├── main.py             # FastAPI: REST API + раздача фронтенда
│   ├── config.py           # пути, список типов БД и правил
│   ├── storage.py          # внутренняя SQLite: подключения, правила, история, настройки
│   ├── i18n.py             # переводы RU/EN
│   ├── scheduler.py        # APScheduler: интервальные задания + логика уведомлений
│   ├── connectors/         # адаптеры БД (base + sqlite/postgresql/clickhouse)
│   ├── rules/engine.py     # движок проверок (6 типов правил)
│   ├── notifiers/          # telegram / email / mattermost
│   └── static/             # фронтенд: index.html + app.js + style.css (без сборки)
├── demo/make_demo.py       # генератор демо-базы + регистрация демо-правил
└── data/sentinel.db        # создаётся автоматически при первом запуске
```

Поток данных: планировщик по интервалу правила вызывает `engine.run_rule()` →
результат (`ok` / `alert` / `error`) пишется в историю → при тревоге и прошедшем
cooldown — рассылка во все включённые каналы; при восстановлении — отдельное уведомление.

## Детали и ограничения

- Если таблица хранит таймстемпы в UTC (частый случай для ClickHouse), включи
  чекбокс «Таймстемпы в таблице в UTC» в правилах «Свежесть» и «Объём» — иначе
  проверка будет ошибаться на величину твоего часового пояса.
- Для ClickHouse за HTTPS включи «HTTPS/TLS-подключение» в форме подключения
  (порт обычно 8443 вместо 8123).
- Если ClickHouse за корпоративным VPN/прокси с перехватом TLS (ошибка
  «SSL: CERTIFICATE_VERIFY_FAILED»), укажи в подключении путь к корневому
  сертификату компании (PEM) — его можно взять у админов или экспортировать
  из браузера. Крайний случай — снять галочку «Проверять SSL-сертификат»,
  но так соединение уязвимо к подмене.
- Тот же корпоративный сертификат лечит SSL-ошибки уведомлений
  (Telegram/Mattermost): задай `REQUESTS_CA_BUNDLE=путь\к\corp-ca.pem`
  перед запуском. Для git: `git config --global http.sslBackend schannel`.
- Сообщения проверок сохраняются в историю на языке, который был активен в момент проверки.
- Для правил с окном времени колонка даты должна сравниваться со строкой
  `YYYY-MM-DD HH:MM:SS` (в SQLite храните таймстемпы в ISO-формате).
- Тип «Аномалия» начинает срабатывать после накопления истории
  (по умолчанию 5 запусков) — до этого пишет «накапливаю историю».
  Если история нужна сразу — используй «Аномалию по истории таблицы».
- В «Аномалии по истории таблицы» пропущенные периоды считаются нулём, поэтому
  она ловит и «данные вообще не приехали». Текущий незавершённый период не
  проверяется (он всегда выглядел бы аномально) — проверяется вчерашний день
  или предыдущий час. Порог в сигмах подбирай по своим данным: для стабильных
  ежедневных загрузок обычно хватает 3σ, для шумных почасовых — 2σ.
- Пароли подключений и токены хранятся в `data/sentinel.db` в открытом виде —
  не выкладывай эту папку в git (см. `.gitignore`).
- Приложение слушает только `127.0.0.1` и не имеет авторизации: для доступа
  извне поставь reverse-proxy с auth.

- Таймер автозапусков отсчитывается от старта приложения и сбрасывается при
  изменении любого правила — время следующего запуска всегда видно на карточке
  правила на дашборде.
- В поле «SQL-источник» пиши только SELECT/WITH-запрос без точки с запятой;
  колонки в остальных полях правила должны существовать в результате этого
  запроса.
